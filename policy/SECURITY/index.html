    高優先度（今すぐやるべき）¶

    1. Rate Limiting — 実装済み¶

    ブルートフォース攻撃・DoS攻撃を防ぐ最初の防衛線。express-rate-limit で実装済み。

    • ログイン: 10回/15分
    • API全体: 200回/15分

    2. Nginxセキュリティヘッダー — 実装済み¶

    ブラウザ側のセキュリティ機能を有効化。EC2上の Nginx に以下を設定済み（2026-04-29）。

    add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;

    3. ファイルアップロード検証強化 — 実装済み¶

    拡張子偽装（.jpg に見せかけたスクリプト等）への対策。MIMEタイプをファイルのバイナリ内容（magic number）から検証する。

    実装内容 (sol/src/lib/sanitize.ts + sol/src/middleware/upload.ts):

    • multer の fileFilter で宣言 MIME タイプを許可リストと照合（第1層）
    • validateUploadedFile() でバッファの magic bytes を検証（第2層）
    • 空ファイル・サイズ超過もここで検出し DB 書き込み前に弾く

    検証対象タイプとマジックバイト:

    MIME タイプ magic bytes
    image/jpeg FF D8 FF
    image/png 89 50 4E 47
    image/webp 52 49 46 46（RIFF）+ offset8 WEBP
    image/gif 47 49 46 38
    video/webm 1A 45 DF A3（EBML）
    video/mp4 offset4 66 74 79 70（ftyp）
    audio/webm 1A 45 DF A3（EBML）
    audio/mpeg（MP3） FF FB / FF F3 / FF F2 / ID3
    audio/wav RIFF + offset8 WAVE

    4. npm audit（既知の脆弱性チェック）— 継続対応中¶

    依存パッケージに既知の脆弱性がないか確認。2026-04-30 最終実施。

    cd sol && npm audit
cd terra && npm audit

    2026-04-30 時点の対応状況（sol）:

    パッケージ 深刻度 内容 対応状況
    xmldom（passport-twitter 経由） critical / high XML パース脆弱性 解消済み: passport-twitter を passport-oauth1 直接実装に置き換え
    drizzle-orm < 0.45.2 high SQL 識別子エスケープ不備（SQL injection） 保留: 0.31→0.45 はメジャー破壊的変更。ユーザー入力を識別子として使用しているコードは存在しないため実質リスク低。次のメジャーアップグレード時に対応
    esbuild ≤ 0.24.2（drizzle-kit 経由） moderate dev server リクエスト露出 対応不要: drizzle-kit は dev-only ツール、本番サーバーには含まれない

    terra は2026-04-30 現在 npm audit で脆弱性ゼロ。

    5. EC2 Security Groupsの確認¶

    外部から直接アクセスできるポートを最小限にする。AWSコンソールで確認。

    許可すべきポート（Inbound）:

    ポート 用途 許可元
    22 SSH 自分のIPのみ（EC2 Instance Connect経由）
    80 HTTP 0.0.0.0/0（HTTPS redirect用）
    443 HTTPS 0.0.0.0/0

    閉じるべきポート: 4000, 4001, 3000, 3001, 5432 は外部から直接アクセス不可にする。

    中優先度（近いうちに）¶

    6. ログイン失敗のアカウントロック — 実装済み¶

    パスワードブルートフォースへの対策。Rate Limiting（IP単位）と組み合わせて2層防御。

    実装内容:

    • 連続失敗5回でアカウントを15分ロック（login_fail_count / login_locked_until カラム）
    • ロック中は ACCOUNT_LOCKED (429) を返し、残り分数をフロントに表示
    • ログイン成功時にカウントを自動リセット
    • migration: 0033_login_lockout.sql、実装: sol/src/api/auth/auth.service.ts

    7. Content Security Policy（CSP）— 実装済み¶

    XSS攻撃の被害を最小化する。悪意あるスクリプトの実行をブラウザ側でブロック。

    実装先: terra/next.config.mjs（async headers() で全ページに適用）

    default-src 'self';
script-src 'self' 'unsafe-inline';   ← Next.js 14 hydration に必要。dev は unsafe-eval も追加
style-src 'self' 'unsafe-inline';    ← Tailwind CSS インラインスタイル
img-src 'self' data: blob: https://plofolia.com https://*.amazonaws.com https://*.cloudfront.net;
connect-src 'self' https://plofolia.com;   ← API・SSE（Nginx 経由で同一オリジン）
media-src 'self' blob: https://plofolia.com https://*.amazonaws.com https://*.cloudfront.net;
worker-src 'self' blob:;   ← Service Worker + MediaRecorder blob
frame-ancestors 'self';
base-uri 'self';
form-action 'self';

    8. JWTリフレッシュトークンのローテーション — 実装済み¶

    使用するたびに新しいリフレッシュトークンを発行し旧トークンを無効化。再使用検出でトークン盗難を検知する。

    実装内容 (sol/src/api/auth/auth.service.ts + migration 0034_refresh_token_rotation.sql):

    • リフレッシュ時に旧トークンを「削除」ではなく「使用済み（used_at）」としてマーク
    • 使用済みトークンが再度提示された場合 → 盗難と判断し対象ユーザーの全セッションを無効化 → TOKEN_REUSE_DETECTED を返す
    • 使用済みトークンは30日後にクリーンアップ（リフレッシュ時に同ユーザーの古い済みトークンを削除）
    • フロント: TOKEN_REUSE_DETECTED 受信時にセキュリティアラートメッセージを表示

    9. セキュリティログの整備 — 実装済み¶

    インシデント発生時の調査・不審なアクセスの事後追跡のために記録する。

    実装内容:

    • security_logs テーブル（migration 0035_security_logs.sql）: event / userId / ip / userAgent / detail / createdAt
    • sol/src/services/securityLogger.ts: logSecurityEvent() ユーティリティ（エラーは握りつぶしてメイン処理を止めない）
    • auth.controller.ts: ログイン成功・失敗・ロック・ログアウト・トークン盗難・パスワードリセット を記録
    • middleware/upload.ts: ファイルタイプ拒否・magic bytes 不一致を記録

    記録イベント一覧:

    イベント トリガー
    login_success ログイン成功
    login_failure パスワード不正
    login_locked アカウントロック発動
    logout ログアウト
    token_reuse_detected 使用済みリフレッシュトークン再提示
    password_reset_request パスワードリセットメール送信
    password_reset_success パスワードリセット完了
    upload_rejected 不正ファイルアップロード

    10. OS自動セキュリティアップデート — 実装済み¶

    EC2（Ubuntu 22.04）にセキュリティパッチを自動適用。

    設定内容（EC2）:

    • unattended-upgrades インストール・有効化済み
    • /etc/apt/apt.conf.d/51plofolia-unattended で追加設定:
    • セキュリティパッチのみ自動適用（通常アップデートは除外）
    • 未使用依存関係の自動削除: true
    • 自動再起動: false（カーネルアップデートは手動再起動）
    # 設定確認
cat /etc/apt/apt.conf.d/20auto-upgrades
cat /etc/apt/apt.conf.d/51plofolia-unattended
# ドライラン
sudo unattended-upgrade --dry-run

    低優先度（将来的に）¶

    11. 二要素認証（2FA）— 実装済み（Aurora MFA + デバイス信頼）¶

    Aurora 管理画面を対象に TOTP MFA を実装済み（一般ユーザー向けは将来対応）。

    実装内容:

    • otplib@12（CJS互換）で TOTP を実装。Google Authenticator / Authy 対応
    • QRコードで設定（/mfa ページ）、有効化・無効化もUIから操作可能
    • ログインフロー: パスワード認証 → 5分有効の pendingToken 発行 → TOTP コード入力で検証
    • デバイス信頼（AWS方式）: MFA 認証後に 64-char hex トークンを発行し SHA-256 ハッシュを admin_trusted_devices に保存（30日間有効）。次回ログイン時に有効なトークンがあれば MFA をスキップ
    • MFA 無効化時は全信頼済みデバイスも削除

    関連ファイル:

    • sol/src/api/admin/admin.auth.ts — ログイン・MFA検証・デバイス信頼ロジック
    • sol/src/db/schema/adminTrustedDevices.ts + migration 0038_admin_trusted_devices.sql
    • aurora/src/app/mfa/page.tsx — MFA設定UI
    • aurora/src/app/login/page.tsx — デバイストークンの localStorage 管理

    12. WAF（Web Application Firewall）¶

    AWS WAFをALB or CloudFrontと組み合わせてSQLインジェクション・XSSパターンをブロック。CloudFront導入（Phase 4）と同時に検討。

    コスト目安: $5/月 + リクエスト数に応じた従量課金

    13. Dependabot（依存関係自動監視）— 実装済み¶

    .github/dependabot.yml を追加済み（2026-04-29）。sol/terra/aurora + GitHub Actions を週次チェック。

    14. ユーザーデータの完全削除¶

    GDPR・忘れられる権利への対応。アカウント削除時に確実に実施する。

    削除対象:

    • DBのユーザーレコード・投稿・DM・いいね・フォロー等（ON DELETE CASCADEで連鎖削除）
    • S3/uploadsのメディアファイル（DBレコードから逆引きして削除）
    • バックアップからの削除は技術的に困難なため、バックアップの保持期間ポリシーで対応

    NSFW制限 年齢確認済みユーザーのみアクセス可

    Plofolia プロダクトロードマップ¶

    サービスとして定めておくべき方針・将来の機能・優先順位をまとめたドキュメント。 インフラのロードマップは ../dev/TECH_STACK.md、セキュリティは ../policy/SECURITY.md を参照。

    決定済み方針¶

    コンテンツガイドライン¶

    詳細は ../policy/CONTENT_POLICY.md を参照。

    カテゴリ 方針
    実写・3DCG・AI生成 OK（AI生成は要表記）
    未成年に見える絵柄の性的描写 絶対NG・例外なし
    グロテスク・ゴア NSFWフラグ必須
    二次創作（版権キャラ） 条件付きOK（成人キャラのみ）
    犯罪を促す内容・doxxing 絶対NG・即BAN

    マネタイズ方針（決定）¶

    Stripe（SFW用）+ CCBill（NSFW用）の二本立て。プラットフォーム手数料10%。

    方針の根拠:

    • Stripeの利用規約はNSFWコンテンツへの利用を制限しており、突然のアカウント停止リスクがある
    • CCBillはOnlyFans等で実績ある成人コンテンツ専用決済
    • 手数料10%はOnlyFans（20%）より低く、Xから移住するクリエイターへの訴求になる
    フェーズ 決済手段 用途 手数料（決済会社） プラットフォーム手数料
    近いうち Stripe SFW扱いのチップ・投げ銭 3.6% + ¥40 10%
    成長期 CCBill 有料サブスク・R18単品販売 9.9% + $0.35 10%

    広告は採用しない。 NSFWコンテンツに広告を配信できる広告ネットワークが限られており、ブランドセーフティの問題からクリエイターへの不信感を招くリスクがある。

    マネタイズロードマップ¶

    Phase 1（現在）: Ko-Fi 寄付のみ¶

    • サービスの認知向上・ユーザー獲得に集中する期間

    Phase 2（近いうち）: クリエイターへのチップ機能¶

    • Stripe でチップ送金機能を実装（SFW扱いのみ）
    • プラットフォーム手数料: 取引の 10%
    • 注意: StripeはNSFWコンテンツへの利用規約が厳しい。突然アカウント停止リスクあり。SFW用途のみStripeとし、NSFWはCCBillで対応

    Phase 3（成長期）: 有料サブスクリプション・単品販売¶

    • 月額フォロワー機能（クリエイターが月額○円のサブスクを設定）
    • コンテンツ単品販売（Pay-per-view）
    • CCBill / Segpay 対応（成人コンテンツ専用決済）

    Phase 4（将来）: プラットフォーム成熟¶

    • クリエイターファンド（人気クリエイターへの還元）
    • 収益レポート・税務書類の自動生成

    コンプライアンス・法務ロードマップ¶

    NSFW特化サービスは一般SNSより法務リスクが高い。

    優先度 項目 内容

    将来 18 U.S.C. § 2257 米国向けにNSFWコンテンツの出演者年齢記録義務（海外展開時）

    チーム・関係者ロードマップ¶

    現在は創業者 + Claude Code の2名体制。ユーザー増加に伴い段階的に関係者を増やす方針。

    Phase 1（現在）: 創業者 + AI¶

    役割 担当 備考
    プロダクト全般 創業者 意思決定・デザイン確認・運営
    コード実装 Claude Code（AI） 実装・コミット・デプロイまで
    モデレーション 創業者 Aurora 管理画面で対応

    Phase 2（ユーザー100〜1,000人）: 外注・副業¶

    • コンテンツモデレーター（副業・委託）: 報告対応・BAN判断。Aurora の閲覧権限を付与。
    • デザイナー（単発委託）: UI改善・ブランドアイデンティティ強化。Figma 共有で対応。
    • 法務アドバイザー（スポット）: 利用規約・プライバシーポリシーのレビュー、DMCA対応フロー構築。

    アクセス権限の原則: 業務に必要な最小権限のみ付与。モデレーターに DB 直接アクセス不要・開発者に本番シークレット不要。

    Phase 3（1,000〜10,000人）: 正社員採用¶

    採用優先度 役割 理由
    最優先 カスタマーサポート / T&S 報告・問い合わせが創業者一人では対応不可能になる
    次 バックエンドエンジニア 決済・スケーリング実装を分担。コードレビュー体制を整える
    次 フロントエンドエンジニア UIの品質・速度向上。デザインシステム構築
    将来 データアナリスト シャドウバン可視化・クリエイター分析ダッシュボードの強化
    将来 法務・コンプライアンス NSFW規制・国際法対応。外注から内製へ

    アクセス管理¶

    追加する関係者 付与するアクセス 付与しないもの
    モデレーター Aurora モデレータアカウント（投稿削除・報告対応・問い合わせ返信） BAN/ユーザー削除・招待コード・監査ログ・GitHub・AWS
    副業エンジニア GitHub（develop ブランチへの PR）・ローカル環境 本番 AWS 認証情報・sol/.env の機密情報
    正社員エンジニア GitHub（全ブランチ）・AWS（必要なサービスのみ） ルートアカウント・支払い情報
    法務アドバイザー ドキュメント（docs/policy/）のみ コード・DB・ユーザーデータ

    AWS IAM 権限設計（エンジニア追加時）¶

    現在は plofolia-admin IAM ユーザー1つで全操作している。エンジニアを追加する場合：

    plofolia-admin（創業者）: EC2・RDS・S3・SES・IAM 全権
plofolia-deploy（CI/CD）: EC2 Instance Connect + SSM のみ
plofolia-engineer（副業）: S3 読み取り + CloudWatch + Cost Explorer のみ

    IAM ユーザーを追加する前に、本番シークレット（JWT_SECRET・ADMIN_JWT_SECRET）を定期ローテーションするポリシーを設けること。

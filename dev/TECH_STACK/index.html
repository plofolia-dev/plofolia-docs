    インフラ可用性ロードマップ¶

    現状の構成（単一EC2）¶

    EC2 t3.micro (ap-northeast-1a)
├── plofolia-sol    (Express, port 4000)
├── plofolia-terra  (Next.js, port 3000)
├── plofolia-nova   (Admin API, port 4001)
├── plofolia-aurora (Admin UI, port 3001)
└── PostgreSQL 16   (port 5432)  ← アプリと同居

    課題: 単一障害点（EC2障害 = 全サービス停止）

    Phase 1 — DBバックアップ（実装済み）¶

    目的: データ消失防止・誤削除からの復元

    EC2 prod ──pg_dump (毎日 AM3:00 JST)──▶ S3 (backups/db/)
                                          └── 30世代保持
EC2 prod ──AMIスナップショット (毎週日曜 AM3:30 JST)──▶ EC2 AMI
                                                          └── 4世代保持

    実装内容:

    • titan/backup-db.sh: pg_dump + gzip → S3 (s3://plofolia-media/backups/db/)、30世代ローテーション
    • titan/backup-ami.sh: EC2 AMI スナップショット（--no-reboot）、4世代ローテーション
    • cron 登録済み（/home/ubuntu/logs/ にログ出力）

    コスト: ~$4/月（AMI EBSスナップショット分のみ）

    DB復元手順:

    # S3からダウンロードして復元
aws s3 cp s3://plofolia-media/backups/db/plofolia_YYYYMMDD_HHMMSS.sql.gz /tmp/
gunzip -c /tmp/plofolia_YYYYMMDD_HHMMSS.sql.gz \
  | PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia

    Phase 2 — ステージングサーバー（ユーザーが増え始めたら）¶

    目的: 本番デプロイ前の動作確認環境

    EC2 prod (t3.micro)          EC2 staging (t3.micro)
plofolia.com                 test.plofolia.com
origin/main ← デプロイ       origin/develop ← 自動デプロイ
PostgreSQL (独立)             PostgreSQL (独立・シードデータ)

    デプロイフロー（改善後）:

    develop ブランチ push
  → CI (GitHub Actions) パス
  → staging 自動デプロイ
  → 動作確認OK
  → main にマージ
  → 本番デプロイ

    コスト: +$8-10/月（EC2 t3.micro 追加）

    Phase 3 — DBレプリケーション（本格稼働期）¶

    目的: DB障害時の自動フェイルオーバー、読み取り負荷分散

    EC2 prod                      EC2 standby
PostgreSQL Primary  ──WAL──▶  PostgreSQL Hot Standby
(Read/Write)                  (Read-Only / 障害時Promote)
     │
     └── repmgr でヘルスチェック・自動昇格管理

    実装内容:

    • PostgreSQL ストリーミングレプリケーション（WAL転送）
    • repmgr による自動ヘルスチェック（10-30秒で検知）
    • 障害時: Standby を自動昇格 → Route 53 DNS 切り替え
    • Standby は通常時も読み取りクエリに活用可能

    設定概要 (postgresql.conf on Primary):

    wal_level = replica
max_wal_senders = 3
wal_keep_size = 64MB

    コスト: +$8-10/月（EC2 t3.micro 追加のみ）

    Phase 4 — 本格HA構成（10万ユーザー規模）¶

    目的: 無停止デプロイ、自動スケーリング、マネージドDB

             Route 53
              │
    ALB（Application Load Balancer）
     ↙                    ↘
EC2 app-1 (t3.small)   EC2 app-2 (t3.small)
     ↘                    ↙
        RDS PostgreSQL Multi-AZ
        Primary (ap-northeast-1a)
             ↕ 自動同期（60秒フェイルオーバー）
        Standby (ap-northeast-1c)
              │
        RDS Read Replica（読み取り専用）

    追加サービス:

    • ALB: ロードバランシング・ヘルスチェック・Blue-Greenデプロイ
    • RDS Multi-AZ: マネージドDB・自動バックアップ・自動フェイルオーバー
    • CloudFront: 静的アセットCDN・画像配信高速化
    • ElastiCache (Redis): セッションキャッシュ・レート制限

    コスト: +$100-150/月（ALB + RDS Multi-AZ + CloudFront）

    ロードマップ サマリー¶

    フェーズ 内容 追加コスト 目安タイミング
    Phase 1 pg_dump→S3日次 + AMIスナップショット週次 ~$4/月 今すぐ
    Phase 2 ステージングサーバー (test.plofolia.com) +$8/月 ユーザーが増え始めたら

    Phase 4 ALB + RDS Multi-AZ + CloudFront + ElastiCache +$100-150/月 10万ユーザー規模

    バックアップとレプリケーションは役割が異なる:

    • バックアップ → 誤削除・データ破損からの復元（時間を遡れる）
    • レプリケーション → サーバー障害時の自動継続（データは最新のまま）
    • 両方を組み合わせることで完全な可用性が実現できる

    技術選定の理由¶

    このプロジェクトの制約条件:

    • 1人体制: 運用・実装どちらも最小コストで回せることが優先
    • t3.micro 予算: 当面のインフラコストは $20-30/月 以内に抑える
    • スピード重視: 機能開発を素早く回せる技術を選ぶ
    • クリエイター特化: 動画・音声・SSE・NSFW など専用要件を満たせること

    フロントエンド¶

    Next.js 14 App Router（vs Vite SPA / Nuxt / Remix）¶

    OGP メタタグによる SNS シェア最適化が必須要件だったため、SSR が前提。 Vite SPA は OGP をサーバーサイドで生成できないため除外。 Nuxt は Vue エコシステムであり、TypeScript + React の知識を活かせる Next.js を選択。 App Router を選んだのは RSC によるサーバーサイドデータフェッチと next-intl の i18n ルーティングが Pages Router より自然に統合できるため。

    Tailwind CSS + Solarized Dark（vs CSS Modules / styled-components）¶

    ユーティリティクラスで UI を素早く組めること、 Solarized Dark のカラーパレットをカスタムトークンとして定義できる柔軟性が決め手。 CSS Modules は命名コストが高く、styled-components は SSR でのハイドレーション設定が煩雑なため除外。

    Zustand（vs Redux Toolkit / Recoil / Jotai）¶

    認証状態・UI 設定（NSFW 表示・言語）を localStorage に永続化する要件に対して、 Redux は boilerplate が多すぎる。Recoil / Jotai は atom 分割設計のコストがかかる。 Zustand は persist ミドルウェアで永続化を数行で実装でき、 バンドルサイズも小さいため採用。

    TanStack Query v5（vs SWR）¶

    無限スクロール（useInfiniteQuery）とカーソルページネーションの組み合わせが タイムライン実装の中心になるため、その API が洗練されている TanStack Query を選択。 SWR は無限スクロールの型サポートが TanStack Query より弱い。

    EventSource / SSE（vs WebSocket）¶

    通知・DM のリアルタイム更新はサーバー→クライアントの一方向で十分。 WebSocket は双方向通信が可能だが、接続管理・スケールアウト時の状態共有が複雑になる。 SSE は HTTP/1.1 の持続接続を使うため、Nginx のリバースプロキシ設定変更が最小限で済む。 また EventSource は認証ヘッダーを付けられないが、?token= クエリパラメータで代替。

    バックエンド¶

    Express（vs Fastify / Hono / NestJS）¶

    選択肢の中で最も枯れており、ミドルウェアのエコシステムが広い。 NestJS はアーキテクチャの学習コストが高く、1人体制では過剰。 Fastify / Hono は高性能だが、Passport.js などのミドルウェアとの相性が Express ほど良くない。 Express はシンプルに書ける分、Claude Code との協業（AI がコードを生成する環境）に向いている。

    Drizzle ORM（vs Prisma / TypeORM）¶

    Prisma は Prisma Client Engine を別プロセスで動かすため、t3.micro では メモリが圧迫される。TypeORM は型安全性が弱い。 Drizzle は軽量でメモリフットプリントが小さく、SQL に近い直感的な API と 完全な TypeScript 型推論を両立できるため採用。

    PostgreSQL（vs MySQL / SQLite）¶

    uuid 型・全文検索・JSONB・配列型など、将来の拡張性が高い。 MySQL も選択肢だが PostgreSQL の方が機能が豊富で、Drizzle との相性も良い。 SQLite は並行書き込みが苦手なため、複数ユーザーが同時投稿する SNS には不適。

    JWT + httpOnly Cookie（vs サーバーセッション / Auth.js）¶

    サーバーセッションはスケールアウト時に Redis 等でセッションストアを共有する必要があり、 単一 EC2 を前提とする現構成では不要な複雑さを加える。 Auth.js（旧 NextAuth）は Next.js に密結合しており、 sol（Express）で独自の認証ロジックを持つ本プロジェクトには合わない。 JWT はステートレスで EC2 インスタンスをまたいでも機能するため、将来の水平スケールにも対応できる。 リフレッシュトークンを httpOnly Cookie に入れることで、XSS によるトークン窃取を防ぐ。

    Zod（vs Yup / Joi / class-validator）¶

    TypeScript との統合が最も深く、スキーマから型を自動推論できる。 Yup は型推論の精度が低い。Joi は TypeScript サポートが後付け。 class-validator はデコレーターを使うため、TypeScript の strict 設定との相性がある。 Zod はスキーマ = 型定義として一元管理でき、フロント・バック間での型共有にも使える。

    インフラ¶

    AWS EC2 t3.micro（vs Vercel / Railway / Fly.io）¶

    Vercel はフロントエンドのホスティングに最適だが、 Express バックエンド・PostgreSQL・ファイルストレージを同じ環境で動かすには 複数サービスを組み合わせる必要があり、コストが跳ね上がる。 Railway / Fly.io はシンプルで良いが、日本リージョン（ap-northeast-1）が選べず 日本ユーザー向けのレイテンシが最適化できない。 EC2 は初期設定コストはあるが、月 $10-15 で 4 プロセス + PostgreSQL を同居できる。 長期的には単価が最も安い。

    Nginx（vs Caddy / Apache）¶

    リバースプロキシとして最も実績があり、設定例が豊富。 Caddy は自動 HTTPS が魅力だが、細かいヘッダー制御の柔軟性は Nginx が上。 Let's Encrypt + Certbot との組み合わせで自動更新も実現できるため、 Caddy の自動 HTTPS メリットが薄れる。

    PM2（vs Docker / systemd）¶

    Docker はコンテナイメージのビルドがメモリを消費し、t3.micro では辛い。 systemd は設定ファイルが煩雑で、複数プロセスのログ集約や再起動ポリシーの管理が手間。 PM2 は Node.js プロセスの管理に特化しており、 pm2 status / pm2 logs など運用コマンドが直感的。

    GitHub Actions（vs CircleCI / GitLab CI）¶

    コードが GitHub にあるため、同一プラットフォームで CI が完結する。 プライベートリポジトリでも月 2,000 分の無料枠があり、現在の規模では費用なし。 PostgreSQL サービスコンテナが標準サポートされており、 インテグレーションテストの環境構築が容易。

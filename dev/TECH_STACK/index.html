    フロントエンド（terra / aurora）¶

    カテゴリ 使用技術 備考
    フレームワーク Next.js 14 (App Router) SSR + RSC
    言語 TypeScript
    スタイリング Tailwind CSS Solarized Dark カスタムカラー（sol.*）
    状態管理 Zustand + zustand/middleware/persist localStorageに認証情報を永続化
    データフェッチ TanStack Query v5 キャッシュ・無限スクロール
    国際化 next-intl ja / en / zh / ko / es の5言語
    アイコン lucide-react
    画像処理 next/image + Canvas API クロップ・エンコード
    メディアエンコード MediaRecorder API + AudioContext API ブラウザ内エンコード（WebM/Opus, WebP）
    HTTP クライアント axios (apiClient) 自動リフレッシュインターセプター付き
    フォーム React 組み込み（controlled components）
    リアルタイム EventSource (SSE) 通知（フォールバック: 60秒）・DM（フォールバック: 30秒）ポーリング
    PWA Service Worker + Web App Manifest public/sw.js + manifest.ts。Android Chrome インストールプロンプト対応

    バックエンド（sol）¶

    カテゴリ 使用技術 備考
    ランタイム Node.js 20
    フレームワーク Express
    言語 TypeScript tsx watch で hot reload
    ORM Drizzle ORM マイグレーション: drizzle-kit
    DB PostgreSQL 16 EC2上にローカルインストール
    認証 JWT アクセス15分 + リフレッシュ7日（httpOnly Cookie）
    OAuth Passport.js + passport-oauth1 Google OAuth 2.0 / Twitter OAuth 1.0a（passport-twitter は xmldom CVE のため廃止・直接実装）
    ファイルアップロード multer ローカルストレージ or S3
    バリデーション Zod
    SSE Express res.write / res.flush 通知・DM
    メール nodemailer プロバイダー切替可（console / smtp / AWS SES）
    DNS検証 dns.promises.resolveMx 登録時のメールドメイン検証
    パスワードハッシュ bcrypt
    アカウントロック DB カラム（login_fail_count / login_locked_until） 失敗5回で15分ロック。フロントに残り分数を表示（ACCOUNT_LOCKED 429）

    インフラ可用性ロードマップ¶

    現状の構成（単一EC2）¶

    EC2 t3.micro (ap-northeast-1a)
├── plofolia-sol    (Express, port 4000)
├── plofolia-terra  (Next.js, port 3000)
├── plofolia-nova   (Admin API, port 4001)
├── plofolia-aurora (Admin UI, port 3001)
└── PostgreSQL 16   (port 5432)  ← アプリと同居

    課題: 単一障害点（EC2障害 = 全サービス停止）

    Phase 1 — DBバックアップ（実装済み）¶

    目的: データ消失防止・誤削除からの復元

    EC2 prod ──pg_dump (毎日 AM3:00 JST)──▶ S3 (backups/db/)
                                          └── 30世代保持
EC2 prod ──AMIスナップショット (毎週日曜 AM3:30 JST)──▶ EC2 AMI
                                                          └── 4世代保持

    実装内容:

    • titan/backup-db.sh: pg_dump + gzip → S3 (s3://plofolia-media/backups/db/)、30世代ローテーション
    • titan/backup-ami.sh: EC2 AMI スナップショット（--no-reboot）、4世代ローテーション
    • cron 登録済み（/home/ubuntu/logs/ にログ出力）

    コスト: ~$4/月（AMI EBSスナップショット分のみ）

    DB復元手順:

    # S3からダウンロードして復元
aws s3 cp s3://plofolia-media/backups/db/plofolia_YYYYMMDD_HHMMSS.sql.gz /tmp/
gunzip -c /tmp/plofolia_YYYYMMDD_HHMMSS.sql.gz \
  | PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia

    Phase 2 — ステージングサーバー（ユーザーが増え始めたら）¶

    目的: 本番デプロイ前の動作確認環境

    EC2 prod (t3.micro)          EC2 staging (t3.micro)
plofolia.com                 test.plofolia.com
origin/main ← デプロイ       origin/develop ← 自動デプロイ
PostgreSQL (独立)             PostgreSQL (独立・シードデータ)

    デプロイフロー（改善後）:

    develop ブランチ push
  → CI (GitHub Actions) パス
  → staging 自動デプロイ
  → 動作確認OK
  → main にマージ
  → 本番デプロイ

    コスト: +$8-10/月（EC2 t3.micro 追加）

    Phase 3 — DBレプリケーション（本格稼働期）¶

    目的: DB障害時の自動フェイルオーバー、読み取り負荷分散

    EC2 prod                      EC2 standby
PostgreSQL Primary  ──WAL──▶  PostgreSQL Hot Standby
(Read/Write)                  (Read-Only / 障害時Promote)
     │
     └── repmgr でヘルスチェック・自動昇格管理

    実装内容:

    • PostgreSQL ストリーミングレプリケーション（WAL転送）
    • repmgr による自動ヘルスチェック（10-30秒で検知）
    • 障害時: Standby を自動昇格 → Route 53 DNS 切り替え
    • Standby は通常時も読み取りクエリに活用可能

    設定概要 (postgresql.conf on Primary):

    wal_level = replica
max_wal_senders = 3
wal_keep_size = 64MB

    コスト: +$8-10/月（EC2 t3.micro 追加のみ）

    Phase 4 — 本格HA構成（10万ユーザー規模）¶

    目的: 無停止デプロイ、自動スケーリング、マネージドDB

             Route 53
              │
    ALB（Application Load Balancer）
     ↙                    ↘
EC2 app-1 (t3.small)   EC2 app-2 (t3.small)
     ↘                    ↙
        RDS PostgreSQL Multi-AZ
        Primary (ap-northeast-1a)
             ↕ 自動同期（60秒フェイルオーバー）
        Standby (ap-northeast-1c)
              │
        RDS Read Replica（読み取り専用）

    追加サービス:

    • ALB: ロードバランシング・ヘルスチェック・Blue-Greenデプロイ
    • RDS Multi-AZ: マネージドDB・自動バックアップ・自動フェイルオーバー
    • CloudFront: 静的アセットCDN・画像配信高速化
    • ElastiCache (Redis): セッションキャッシュ・レート制限

    コスト: +$100-150/月（ALB + RDS Multi-AZ + CloudFront）

    ロードマップ サマリー¶

    フェーズ 内容 追加コスト 目安タイミング
    Phase 1 pg_dump→S3日次 + AMIスナップショット週次 ~$4/月 今すぐ
    Phase 2 ステージングサーバー (test.plofolia.com) +$8/月 ユーザーが増え始めたら
    Phase 3 PostgreSQLストリーミングレプリケーション + repmgr +$8/月 本格稼働期
    Phase 4 ALB + RDS Multi-AZ + CloudFront + ElastiCache +$100-150/月 10万ユーザー規模

    バックアップとレプリケーションは役割が異なる:

    • バックアップ → 誤削除・データ破損からの復元（時間を遡れる）
    • レプリケーション → サーバー障害時の自動継続（データは最新のまま）
    • 両方を組み合わせることで完全な可用性が実現できる

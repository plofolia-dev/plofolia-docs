    開発フロー¶

    ブランチ構成¶

    ブランチ 用途
    develop 普段の開発・CI（GitHub Actions でテスト自動実行）
    main 本番サーバーが参照する

    通常の開発サイクル¶

    1. 実装（Claude が担当）¶

    develop ブランチでコードを変更・コミット・push まで行う。

    git checkout develop
# コード変更 → commit → push origin develop

    2. 動作確認（ユーザーが確認）¶

    ローカルの dev サーバー（npm run dev:app）または Cloudflare Tunnel 経由で確認。 問題なければデプロイ承認を Claude に伝える。

    3. デプロイ（Claude が実行・ユーザーが承認後）¶

    # ローカル（PowerShell）からワンコマンドで完結
.\titan\deploy.ps1

    詳細は ../ops/DEPLOY.md を参照。

    役割分担¶

    開発フロー¶

    作業 担当
    コード変更・コミット Claude
    git push origin develop Claude（自律実行）
    ローカル動作確認・デプロイ承認 ユーザー
    .\titan\deploy.ps1（EC2デプロイ） Claude（ユーザー承認後）
    main へのマージ Claude（ユーザー承認後）

    判断・意思決定¶

    作業 担当 備考
    機能追加・削除の判断 ユーザー Claude はオプションを提案するが決定はユーザー
    デザイン・UI の最終承認 ユーザー Claude が実装・Cloudflare Tunnel で確認してもらう
    コンテンツポリシー・利用規約の判断 ユーザー Claude は草案を作るが法的判断はユーザー
    BAN・モデレーション判断 ユーザー 現フェーズは Aurora 管理画面でユーザーが対応
    EC2 デプロイ実行 Claude ユーザーが「デプロイOK」と言った後に自律実行
    DB マイグレーション実行（本番） Claude ユーザー承認後に SSH で実行
    AWS リソースの作成・削除 Claude 費用に影響するため事前に内容を報告してから実行

    Claude が自律実行してよいこと（確認不要）¶

    作業 備考
    ローカルでのコード変更・ファイル編集 可逆的な操作
    git commit / git push origin develop develop ブランチへの push のみ
    EC2 での Nginx reload・PM2 restart ダウンタイムが数秒以内
    ドキュメント更新 コードに影響しない
    npm audit / ログ確認 読み取りのみ

    Claude が必ずユーザーに確認してから実行すること¶

    作業 理由
    main ブランチへのマージ・push 本番に影響するため
    DB スキーマ変更（本番マイグレーション） データ消失リスク
    AWS リソースの新規作成・削除 費用・アーキテクチャに影響
    EC2 インスタンスの停止・再起動 サービス停止を伴う
    外部サービスの設定変更（DNS・SSL等） 本番サービスに即時影響
    .env などの機密情報変更 セキュリティリスク

    コミットメッセージの書き方¶

    feat:     新機能追加
fix:      バグ修正
docs:     ドキュメント更新
refactor: リファクタリング
style:    デザイン変更
chore:    設定・依存関係の変更

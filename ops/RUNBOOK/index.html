    Plofolia 障害対応・トラブルシューティング¶

    障害発生時の初動・復旧手順と、症状別の開発者向けトラブルシューティングをまとめたドキュメント。
    パニックになっても上から順に読めば動けるよう設計してある。

    1. 重大度レベル（P1〜P4）¶

    レベル 定義 例 目標復旧時間
    P1 サービス全体が使えない、またはデータ消失・漏洩リスク 全面ダウン / DB破損 / 不正アクセス 1時間以内
    P2 一部機能が停止しているが主要フローは動いている 1サービスのみ停止 / 画像表示不可 / 通知届かない 4時間以内
    P3 機能は動いているが品質・信頼性が低下している デプロイ失敗 / レスポンス遅延 / エラー率上昇 24時間以内
    P4 軽微な問題。ユーザー影響が最小限または限定的 特定ページのレイアウト崩れ / 翻訳キー欠落 次回デプロイ

    2. 初動対応フロー¶

    障害を検知したら最初にこの手順を実行する。

    1. 重大度を判断（上の表を参照）
2. EC2 に SSH 接続（下記の共通手順を参照）
3. pm2 status → 異常なサービスを特定
4. pm2 logs <service> --lines 100 --nostream → エラー原因を確認
5. 当てはまるシナリオの手順へ進む

    SSH 接続（ローカル PowerShell から）¶

    $aws = "C:\Program Files\Amazon\AWSCLIV2\aws.exe"
$pubKey = Get-Content "E:\DEV\plofolia\ec2_key.pub" -Raw
& $aws ec2-instance-connect send-ssh-public-key `
  --instance-id i-04f92f1a7aad49ef2 `
  --availability-zone ap-northeast-1a `
  --instance-os-user ubuntu `
  --ssh-public-key $pubKey
ssh -i E:\DEV\plofolia\ec2_key -o StrictHostKeyChecking=no ubuntu@13.115.254.4

    状態確認コマンド（EC2 上）¶

    pm2 status                                      # サービス一覧
pm2 logs --lines 50 --nostream                  # 全サービスの直近ログ
sudo systemctl status nginx                     # Nginx の状態
sudo systemctl status postgresql                # PostgreSQL の状態
df -h                                           # ディスク使用量
free -h                                         # メモリ使用量
curl -s http://127.0.0.1:4000/health            # sol ヘルスチェック

    3. P1: 全サービス停止¶

    https://plofolia.com に到達できない。

    Step 1: EC2 インスタンスの確認¶

    AWS コンソール → EC2 → インスタンス i-04f92f1a7aad49ef2 のステータスを確認。

    • 停止中 → 「インスタンスを起動」で起動。起動後 2〜3 分待ってから再確認。
    • 実行中 → Step 2 へ。

    Step 2: SSH 接続できるか確認¶

    上記の SSH 接続コマンドを実行。接続できなければ:

    • セキュリティグループでポート 22 が開いているか AWS コンソールで確認
    • EC2 Instance Connect でブラウザから接続を試みる（AWS コンソール → 接続 → EC2 Instance Connect）

    Step 3: PM2 サービスを再起動¶

    pm2 status
pm2 restart all
pm2 status   # 全サービスが online になるまで待つ

    それでも起動しないサービスがあれば:

    pm2 logs plofolia-sol --lines 100 --nostream
# エラー内容を確認して下のシナリオへ進む

    Step 4: Nginx を確認・再起動¶

    sudo systemctl status nginx
sudo nginx -t           # 設定構文チェック
sudo systemctl reload nginx

    Step 5: ディスクフル¶

    df -h
# /dev/root が 95% 以上なら空き確保
pm2 flush                         # PM2 ログをクリア（数十〜数百MB空く場合がある）
sudo journalctl --vacuum-size=100M  # systemd ジャーナルを 100MB に圧縮
find /tmp -name "*.gz" -mtime +1 -delete  # 古い一時ファイルを削除
npm cache clean --force

    Step 6: メモリ不足（OOM Kill）¶

    free -h
dmesg | grep -i "oom\|killed"   # OOM Killer の発動ログを確認

    OOM Kill が発生していた場合:

    # swap が有効か確認
swapon --show
# swap がなければ作成（2GB 推奨）
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
# 永続化
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

    PM2 を再起動してメモリ使用量を監視:

    pm2 restart all
watch -n 5 'free -h && pm2 status'

    4. P1: DB 破損 / データ消失¶

    ログに relation does not exist / database is corrupt / FATAL: could not open file などが出る。
    または誤った SQL で大量データを削除してしまった場合。

    Step 1: 現状確認¶

    sudo systemctl status postgresql
PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "SELECT COUNT(*) FROM users;"

    接続できない場合はまず PostgreSQL を再起動:

    sudo systemctl restart postgresql

    Step 2: バックアップ一覧を確認¶

    aws s3 ls s3://plofolia-media/backups/db/ | sort | tail -10

    Step 3: リストア（pg_dump 方式）¶

    aws s3 cp s3://plofolia-media/backups/db/plofolia_YYYYMMDD_HHMMSS.sql.gz /tmp/

pm2 stop plofolia-sol plofolia-nova

PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia \
  -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
gunzip -c /tmp/plofolia_YYYYMMDD_HHMMSS.sql.gz \
  | PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia

cd ~/plofolia && npm run db:migrate   # リストア後にスキーマが古い場合

pm2 restart plofolia-sol plofolia-nova
pm2 status

    Step 4: 特定時点へのロールバック（PITR）¶

    WAL アーカイビングを有効化している場合のみ対応可能（OPERATIONS.md #11 参照）。

    sudo systemctl stop postgresql

sudo cp -r /var/lib/postgresql/16/main /var/lib/postgresql/16/main_before_recovery

aws s3 ls s3://plofolia-media/pgbase/ | sort | tail -5
aws s3 cp s3://plofolia-media/pgbase/base_YYYYMMDD.tar.gz /tmp/

sudo rm -rf /var/lib/postgresql/16/main
sudo mkdir /var/lib/postgresql/16/main
sudo tar -xzf /tmp/base_YYYYMMDD.tar.gz -C /var/lib/postgresql/16/main
sudo chown -R postgres:postgres /var/lib/postgresql/16/main

sudo -u postgres bash -c "cat >> /var/lib/postgresql/16/main/postgresql.conf << 'EOF'
restore_command = 'aws s3 cp s3://plofolia-media/wal-archive/%f %p'
recovery_target_time = '2026-04-30 14:31:00'   # 復元したい時刻（JST）
recovery_target_action = 'promote'
EOF"

sudo systemctl start postgresql
sudo tail -f /var/log/postgresql/postgresql-16-main.log

pm2 restart plofolia-sol plofolia-nova

    重要: リカバリ後は postgresql.conf からリカバリ設定を削除し sudo systemctl reload postgresql を実行すること。

    Step 5: EC2 全体を失った場合¶

    MIGRATION.md の「EC2 インスタンス移行」を参照。

    5. P1: セキュリティ侵害¶

    不正アクセス・データ漏洩・アカウント乗っ取りが疑われる場合。

    Step 1: 被害を広げない（封じ込め）¶

    # セキュリティグループで全インバウンドを即時遮断（AWS コンソール）
# → EC2 → セキュリティグループ → インバウンドルールを全削除
# ※ SSH も遮断されるため事前に作業を完了させること

# または Nginx を停止してサービスのみ遮断（SSH は維持）
sudo systemctl stop nginx

    Step 2: セキュリティログを確認¶

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "
SELECT event, user_id, ip, user_agent, created_at, detail
FROM security_logs
WHERE created_at > NOW() - INTERVAL '24 hours'
ORDER BY created_at DESC LIMIT 100;"

sudo grep "Failed\|Accepted\|Invalid" /var/log/auth.log | tail -50

sudo tail -200 /var/log/nginx/access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -20

    Step 3: 影響範囲の特定¶

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "
SELECT username, email, role, created_at FROM users
WHERE role IN ('admin','moderator') ORDER BY created_at DESC;"

find ~/plofolia -name "*.js" -newer ~/plofolia/package.json -not -path "*/node_modules/*" -not -path "*/.next/*"

    Step 4: 全セッションを無効化¶

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "
DELETE FROM refresh_tokens;"

# 管理者パスワード変更（Aurora → アカウント管理 から）
# 緊急時のみ DB で直接更新:
# node -e "const b=require('bcrypt'); b.hash('新パスワード',10).then(h=>console.log(h))"
# UPDATE users SET password_hash='ハッシュ' WHERE username='admin';

    Step 5: 不審なアカウントを BAN¶

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "
UPDATE users SET is_banned = true WHERE username = '不審なアカウント名';"

    Step 6: Nginx の IP ブロック¶

    sudo nano /etc/nginx/sites-available/plofolia
# server ブロック内に追加: deny 攻撃元IP;
sudo nginx -t && sudo systemctl reload nginx

    Step 7: パスワード・シークレット変更¶

    侵害が確認された場合:

    1. EC2 SSH キーペアを再生成（AWS コンソール → EC2 → キーペア）
    2. JWT_SECRET を変更（sol/.env → pm2 restart --update-env）
    3. DB パスワードを変更（ALTER USER plofolia WITH PASSWORD '新パスワード'）
    4. AWS IAM アクセスキーをローテーション（コンソール → IAM → セキュリティ認証情報）

    Step 8: インシデント記録¶

    ポストモーテムテンプレート を使って記録する。

    6. P2: 単一サービス停止¶

    1 サービスだけが errored / stopped になっている場合。

    確認・復旧¶

    pm2 status
pm2 logs plofolia-sol --lines 100 --nostream
pm2 restart plofolia-sol

    よくある原因と対処¶

    ログのキーワード 原因 対処
    EADDRINUSE: address already in use 前のプロセスが残っている lsof -i :4000 \| grep LISTEN → kill <PID> → pm2 restart
    Cannot find module npm install 未実施 cd ~/plofolia/sol && npm install && pm2 restart plofolia-sol
    ECONNREFUSED 127.0.0.1:5432 PostgreSQL が停止 sudo systemctl start postgresql
    password authentication failed DB 認証情報の不一致 cat ~/plofolia/sol/.env で DATABASE_URL を確認
    relation "xxx" does not exist マイグレーション未適用 OPERATIONS.md #5「マイグレーションの適用」参照
    JavaScript heap out of memory メモリ不足（t3.micro） swap を増量（→ P1 全面ダウン Step 6 参照）
    SyntaxError / TypeError (起動直後) TypeScript ビルドが古い cd sol && npm run build && pm2 restart plofolia-sol

    terra が起動しない場合¶

    pm2 logs plofolia-terra --lines 100 --nostream
cd ~/plofolia/terra
rm -rf .next
NODE_OPTIONS="--max-old-space-size=1500" npm run build
pm2 restart plofolia-terra --update-env

    7. P2: メディアファイル消失¶

    画像・動画・音声が 404 で表示されない。uploads/ ディレクトリが空。

    Step 1: 現状確認¶

    ls -lh ~/plofolia/sol/uploads/
du -sh ~/plofolia/sol/uploads/

    Step 2: S3 バックアップから復元¶

    aws s3 ls s3://plofolia-media/media-backups/ --recursive --human-readable --summarize | tail -3

aws s3 sync s3://plofolia-media/media-backups/ ~/plofolia/sol/uploads/

curl -I https://plofolia.com/api/v1/uploads/images/XXXXXX.jpg
# 200 が返れば正常

    8. P3: デプロイ失敗・ロールバック¶

    titan/deploy.ps1 が途中で止まった¶

    .\titan\deploy.ps1

    よくある失敗パターン:

    公開鍵プッシュのタイムアウト（60秒制限）
    → キーを再プッシュして再実行。スクリプトは冪等なので再実行可能。

    git push が rejected

    git pull origin develop --rebase
git push origin develop

    terra ビルドがメモリ不足でクラッシュ（t3.micro）

    free -h
swapon --show

pm2 stop all
cd ~/plofolia/terra
NODE_OPTIONS='--max-old-space-size=750' npm run build
pm2 start all
pm2 save

    ポイント: サービスを停止してからビルドすること。起動中のビルドはメモリ競合で OOM になることがある。

    DBマイグレーションを適用するのを忘れた

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia \
  -f ~/plofolia/sol/src/db/migrations/<ファイル名>.sql
pm2 restart plofolia-sol

    パターン A: git で前のコミットに戻す¶

    cd ~/plofolia
git log --oneline -5
git reset --hard <コミットハッシュ>

cd sol && npm install && npm run build && cd ..
pm2 restart plofolia-sol plofolia-nova

# terra も戻す場合
cd terra && NODE_OPTIONS="--max-old-space-size=1500" npm run build && cd ..
pm2 restart plofolia-terra --update-env

    ローカルからも revert:

    git revert HEAD
git push origin develop
.\titan\deploy.ps1

    パターン B: DB マイグレーションを誤って適用した¶

    cat ~/plofolia/sol/src/db/migrations/<ファイル名>.sql

PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia << 'EOF'
-- 例: カラムを追加していた場合
ALTER TABLE users DROP COLUMN IF EXISTS new_column;
EOF

PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "
DELETE FROM drizzle.__drizzle_migrations WHERE hash = '...対象ハッシュ...';"

    教訓: マイグレーションは本番適用前にローカルで必ず検証すること。破壊的なマイグレーション（カラム削除・型変更）は特に慎重に。

    パターン C: AMI スナップショットから完全リストア¶

    AWS コンソールで実行:
1. EC2 → AMI → 最新の plofolia-ami-YYYYMMDD を選択
2. 「AMIからインスタンスを起動」
3. 同じインスタンスタイプ（t3.micro）・セキュリティグループ・キーペアで起動
4. 起動後に Elastic IP を新インスタンスに付け替える
   ※ EIP 未使用の場合は Route 53 や Nginx の upstream を変更する

    9. P3: 過負荷 / DDoS¶

    現状確認¶

    top
htop   # sudo apt install htop

ss -s
sudo netstat -ntp | grep :443 | wc -l

sudo tail -1000 /var/log/nginx/access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -20

    特定 IP からの攻撃¶

    sudo nano /etc/nginx/sites-available/plofolia
# server ブロック外（http ブロック内）に: deny 攻撃元IP;
sudo nginx -t && sudo systemctl reload nginx

    過負荷でサービスが応答しない¶

    pm2 stop plofolia-terra   # フロントエンドを一時停止（API は継続）
# 原因調査後に再起動
pm2 start plofolia-terra

    Rate Limiting の一時強化¶

    nano ~/plofolia/sol/.env
# RATE_LIMIT_MAX=50  ← デフォルト 200 を下げる
pm2 restart plofolia-sol --update-env

    10. 症状別トラブルシューティング¶

    10-1. 502 Bad Gateway¶

    Nginx は動いているがバックエンドに接続できない状態。

    Step 1: ポートが listen しているか確認

    curl -s http://127.0.0.1:4000/health   # → {"status":"ok"} でなければ異常
curl -s http://127.0.0.1:3000          # → HTML が返れば正常
curl -s http://127.0.0.1:4001/health

    Step 2: PM2 ログで原因特定

    pm2 logs plofolia-sol --lines 100 --nostream
pm2 logs plofolia-terra --lines 100 --nostream

    Step 3: よくある原因と対処

    症状 原因 対処
    sol が起動失敗 npm run build されていない cd sol && npm run build && pm2 restart plofolia-sol
    terra が起動失敗 --cwd なしで起動 pm2 delete plofolia-terra && pm2 start ... --cwd /home/ubuntu/plofolia/terra
    環境変数が反映されない --update-env なし pm2 restart plofolia-sol --update-env
    Nginx proxy 先ポートが違う nginx 設定ミス sudo cat /etc/nginx/sites-enabled/plotia でポート確認

    Step 4: Nginx エラーログ確認

    sudo tail -50 /var/log/nginx/error.log

    10-2. React Hydration エラー（#418/#425/#423）¶

    ブラウザコンソールに Hydration failed / Text content does not match が出る、またはページが白くなる。

    原因: Zustand の persist が localStorage から auth 状態を復元するタイミングと、SSR のタイミングがずれる。

    修正パターン（実装済み）

    const { isAuthenticated, _hasHydrated } = useAuthStore()

// NG: SSR と CSR で値が異なり hydration mismatch になる
{isAuthenticated() && <AuthOnlyUI />}

// OK: hydration 完了後のみ条件付きレンダリング
{_hasHydrated && isAuthenticated() && <AuthOnlyUI />}

    または mounted パターン（Header など）:

    const [mounted, setMounted] = useState(false)
useEffect(() => setMounted(true), [])
const authed = mounted && isAuthenticated()

    ブラウザキャッシュが原因の場合: Ctrl+Shift+R または DevTools → Application → Clear site data で解消。

    10-3. 翻訳キー欠落クラッシュ（MISSING_MESSAGE）¶

    ページが白くなり、PM2 ログに MISSING_MESSAGE: xxx.yyy (ja) が出る。

    Step 1: ログで欠落キーを特定

    pm2 logs plofolia-terra --lines 50 --nostream | grep MISSING

    Step 2: 該当キーを全 5 言語ファイルに追加

    terra/messages/ 以下の ja.json / en.json / zh.json / ko.json / es.json。
    重要: 1 言語でもキーが欠けていると、その言語でクラッシュする。必ず 5 ファイル全て更新すること。

    Step 3: 修正後にデプロイ

    .\titan\deploy.ps1

    10-4. ログイン・認証の不具合¶

    「ログインできない」

    1. メール認証が完了しているか確認（未認証ユーザーはログイン不可）
    2. Aurora で確認: https://aurora.plofolia.com → ユーザー一覧 → 「メール未認証」フィルタ → 「メール認証を手動で完了させる」

    緊急時（DB 直接操作）:

    -- 状態確認
SELECT username, email, email_verified_at, email_verification_token
FROM users WHERE email = 'xxx@example.com';

-- 手動認証（Aurora から操作できない場合のみ）
UPDATE users SET email_verified_at = NOW(), email_verification_token = NULL
WHERE email = 'xxx@example.com';

    JWT が期限切れで弾かれる

    • アクセストークン: 15 分で期限切れ → リフレッシュトークン（7 日）で自動更新
    • リフレッシュも切れた場合は再ログインが必要（仕様通り）

    OAuth（Google/Twitter）ログインが失敗する

    • Google コンソールのコールバック URL に https://plofolia.com/api/v1/oauth/google/callback が登録されているか確認
    • Twitter コンソールも同様に確認

    10-5. ファイルアップロードエラー¶

    400 Bad Request が返る

    • 動画 MIME タイプに codecs パラメータが含まれていないか確認（video/webm; codecs=vp9 → video/webm に変換済み）
    • ファイル数制限（画像 4 枚、動画 1 本のみ、音声+カバー画像 1 枚）を超えていないか

    アップロードしたファイルが表示されない

    ls -la ~/plofolia/sol/uploads/

    STORAGE_PROVIDER=local の場合、ファイルは EC2 の sol/uploads/ に保存される。

    10-6. SSE（リアルタイム通知/DM）が届かない¶

    1. ブラウザ DevTools の Network タブ → stream?token= で始まるリクエストが pending になっているか確認
    2. 25 秒ごとに keepalive（: keepalive）が流れているか確認
    3. SSE 接続が失敗している場合、フロントは 60 秒ポーリングにフォールバックする（通知は遅れるが届く）

    Nginx の SSE 設定確認

    sudo grep -A5 'stream' /etc/nginx/sites-enabled/plotia

    必要なディレクティブ:

    proxy_buffering off;
proxy_cache off;
proxy_read_timeout 3600s;

    10-7. 管理画面（Aurora）に入れない¶

    アクセス手順

    https://aurora.plofolia.com で直接アクセス可能（SSH トンネル不要）。

    1. HTTP Basic Auth ダイアログ: ユーザー名 aurora / パスワードは CLAUDE.local.md 参照
    2. Aurora ログイン画面: ユーザー名 admin / パスワード Password123!

    JWT は 15 分で失効。操作中に認証エラーが出たら再ログインする。

    「HTTP Basic Auth ダイアログが繰り返し出る」

    Nginx レベルの認証失敗。CLAUDE.local.md に記載のパスワードを正確に入力する。
    パスワードを変更した場合は EC2 上で htpasswd ファイルを再生成:

    sudo htpasswd -bc /etc/nginx/.htpasswd_aurora aurora '新しいパスワード'
sudo systemctl reload nginx

    「Invalid credentials」でログインできない

    admin アカウントが存在するか確認:

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia \
  -c "SELECT username, role, email_verified_at FROM users WHERE role = 'admin';"

    11. メンテナンスモード¶

    Nginx レベルでメンテナンスページ（静的 HTML）を有効にする方法。

    有効化（計画メンテナンス時）¶

    sudo touch /var/www/plofolia-maintenance/enabled
sudo systemctl reload nginx

    解除¶

    sudo rm /var/www/plofolia-maintenance/enabled
sudo systemctl reload nginx

    自動フォールバック（Next.js / API が落ちた場合）¶

    設定変更不要。Nginx が 502/503/504 エラーを検知すると /var/www/plofolia-maintenance/maintenance.html を自動で返す。

    メンテナンスページの HTML 更新¶

    scp -i E:/DEV/plofolia/ec2_key titan/maintenance.html ubuntu@13.115.254.4:/tmp/
ssh -i E:/DEV/plofolia/ec2_key ubuntu@13.115.254.4 \
  "sudo cp /tmp/maintenance.html /var/www/plofolia-maintenance/maintenance.html"

    1. サービス状態の確認¶

    pm2 status

    4つのサービスが全て online であれば正常:

    PM2名 ポート 役割
    plofolia-sol 4000 メインAPI
    plofolia-nova 4001 管理API
    plofolia-terra 3000 フロントエンド
    plofolia-aurora 3001 管理画面

    ヘルスチェック（EC2上）¶

    curl -s http://127.0.0.1:4000/health   # → {"status":"ok"}
curl -s http://127.0.0.1:4001/health   # → {"status":"ok"}

    外部からのチェック¶

    curl -s https://plofolia.com/api/v1/health

    2. ログの確認¶

    リアルタイムログ（全サービス）¶

    pm2 logs

    特定サービスのログ（直近100行）¶

    pm2 logs plofolia-sol --lines 100 --nostream
pm2 logs plofolia-terra --lines 100 --nostream
pm2 logs plofolia-nova --lines 100 --nostream
pm2 logs plofolia-aurora --lines 100 --nostream

    Nginxアクセスログ / エラーログ¶

    sudo tail -100 /var/log/nginx/access.log
sudo tail -50 /var/log/nginx/error.log

    ログのクリア（ディスク節約）¶

    pm2 flush   # PM2の全ログをクリア

    3. コードのデプロイ¶

    通常のデプロイ（推奨）¶

    ローカルでコードを確認後、ワンコマンドでデプロイ:

    # ローカル（PowerShell）から実行
.\titan\deploy.ps1

    スクリプトが自動で行うこと:

    1. EC2に一時公開鍵をプッシュ（60秒有効）
    2. git push origin develop
    3. EC2上で git fetch && git reset --hard origin/develop
    4. 変更があったサービスのみビルド・PM2再起動
    5. terra には --update-env を付与

    初回のみ: E:\DEV\plofolia\ec2_key が存在しない場合、スクリプトが自動生成する。

    手動でEC2上から実行する場合¶

    ssh -i /e/DEV/plofolia/ec2_key ubuntu@13.115.254.4
cd ~/plofolia
git fetch origin && git reset --hard origin/develop
./titan/deploy.sh

    特定サービスのみ再起動¶

    pm2 restart plofolia-sol --update-env
pm2 restart plofolia-terra --update-env
pm2 restart all

    4. DBメンテナンス¶

    EC2上でpsqlに接続¶

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia

    よく使うクエリ¶

    -- ユーザー数
SELECT COUNT(*) FROM users;

-- 直近の投稿
SELECT id, user_id, created_at FROM posts ORDER BY created_at DESC LIMIT 10;

-- openなバグ報告
SELECT id, category, status, LEFT(description, 80), created_at
FROM bug_reports WHERE status = 'open' ORDER BY created_at DESC;

-- ユーザー詳細確認
SELECT id, username, email, role, email_verified_at FROM users
WHERE username = 'xxx';

    ローカルのDockerDBに接続¶

    docker exec -i plofolia_db psql -U plofolia -d plofolia

    5. マイグレーションの適用¶

    開発時（ローカル）¶

    cd sol
npm run db:generate   # schema変更後にSQLファイルを生成
npm run db:migrate    # ローカルDBに適用

    本番EC2への適用¶

    ssh -i /e/DEV/plofolia/ec2_key ubuntu@13.115.254.4

# 特定ファイルを適用
PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia \
  -f ~/plofolia/sol/src/db/migrations/<ファイル名>.sql

# 全ファイルを順番に適用（初回セットアップ時など）
for f in ~/plofolia/sol/src/db/migrations/*.sql; do
  echo "Applying $f..."
  PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -f "$f"
done

    マイグレーション適用後は sol を再起動:

    pm2 restart plofolia-sol

    6. バグ報告の対応¶

    Claude Code からの確認（ローカルDB）¶

    npm run bugs            # openのみ
npm run bugs all        # 全ステータス
npm run bugs resolved   # 解決済み
node titan/bugs.js <UUID>  # 特定バグの詳細

    EC2本番DBのバグ確認¶

    ssh -i /e/DEV/plofolia/ec2_key ubuntu@13.115.254.4
PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia \
  -c "SELECT id, category, status, LEFT(description,80) FROM bug_reports WHERE status='open' ORDER BY created_at DESC;"

    Aurora（管理画面）での対応¶

    1. SSHトンネルを張る（→ 管理画面へのアクセス 参照）
    2. http://localhost:3001 にアクセス
    3. 「バグ報告」メニュー → ステータス変更・管理者メモを記入

    7. SSL証明書の更新¶

    証明書は Let's Encrypt で90日有効。Certbotが自動更新するため通常は手動対応不要。

    有効期限の確認¶

    ssh -i /e/DEV/plofolia/ec2_key ubuntu@13.115.254.4
sudo certbot certificates

    現在の有効期限: 2026-07-26

    手動更新（期限が近い場合）¶

    sudo certbot renew
sudo systemctl reload nginx

    自動更新のテスト¶

    sudo certbot renew --dry-run

    8. 管理画面（Aurora）へのアクセス¶

    Aurora は HTTPS で公開済み。SSHトンネル不要。

    アクセス方法¶

    1. URL: https://aurora.plofolia.com
    2. HTTP Basic Auth ダイアログが出たら:
    3. ユーザー名: aurora
    4. パスワード: wE3aioEqMBk36iVqDLCNKLc+（CLAUDE.local.md 参照）
    5. Aurora ログイン画面:
    6. ユーザー名またはメール: admin
    7. パスワード: Password123!（admin アカウント）
    8. MFA 設定済みの場合は認証アプリの6桁コードを入力

    JWTは 15分で失効（自動更新なし）。操作中に「ログインしてください」が出たら再ログイン。

    Aurora でできること¶

    メニュー 内容 権限

    ユーザー 検索・BAN済み/メール未認証フィルタ・ユーザー詳細 admin/mod
    ユーザー詳細 投稿一覧・BAN/解除・削除・メール手動認証 admin のみ
    投稿 投稿一覧・削除 admin/mod
    投稿報告 報告一覧・メディアプレビュー・対応・却下 admin/mod
    BAN異議申し立て 一覧・承認（アンバン）・却下 admin/mod
    バグ報告・要望 一覧・ステータス変更・返信 admin/mod
    お問い合わせ スレッド表示・返信 admin/mod
    招待コード 発行・管理 admin のみ
    操作ログ 全管理操作の履歴 admin のみ
    アカウント管理 モデレーターアカウント作成・削除・ロール変更 admin のみ
    MFA設定 TOTP 設定・信頼済みデバイス管理 admin/mod

    方法2: SSHで直接接続¶

    # 1. 一時公開鍵をプッシュ（PowerShellで）
$aws = "C:\Program Files\Amazon\AWSCLIV2\aws.exe"
$pubKey = Get-Content "E:\DEV\plofolia\ec2_key.pub" -Raw
& $aws ec2-instance-connect send-ssh-public-key `
  --instance-id i-04f92f1a7aad49ef2 `
  --availability-zone ap-northeast-1a `
  --instance-os-user ubuntu `
  --ssh-public-key $pubKey

# 2. Bashから接続

    SSH鍵ファイル: E:\DEV\plofolia\ec2_key（初回は deploy.ps1 が自動生成）

    11. バックアップ戦略¶

    SNSにおけるバックアップの考え方¶

    バックアップ対象はDBとメディアファイルの2種類あり、役割も異なる。

    Plofolia のデータ
├── PostgreSQL DB
│   ├── [最重要] users, posts, follows, DMs, series
│   ├── [重要]   likes, bookmarks, notifications
│   └── [低]     view_count などの集計値（再計算可）
│
└── メディアファイル (uploads/)        ← 現在EC2ローカル
    └── [最重要] ユーザーの作品（画像・動画・音声）
                   失ったら二度と戻らない・信頼問題に直結

    重要: pg_dump はDB内のURL文字列しかバックアップしない。 ファイル本体（uploads/）は別途バックアップが必要。

    Phase 1（現状推奨）: pg_dump + メディアrsync¶

    DBバックアップスクリプト¶

    EC2上の /home/ubuntu/backup_db.sh として設置:

    #!/bin/bash
DATE=$(date +%Y%m%d_%H%M%S)
FILE="plofolia_backup_${DATE}.sql.gz"

PGPASSWORD=polotiaakplt pg_dump -h localhost -U plofolia -d plofolia \
  | gzip > /tmp/${FILE}

aws s3 cp /tmp/${FILE} s3://plofolia-media/db-backups/${FILE}
rm /tmp/${FILE}

# 30世代より古いものを削除
aws s3 ls s3://plofolia-media/db-backups/ \
  | awk '{print $4}' | sort | head -n -30 \
  | xargs -I{} aws s3 rm s3://plofolia-media/db-backups/{}

echo "[$(date)] DB backup done: ${FILE}"

    メディアファイルバックアップスクリプト¶

    EC2上の /home/ubuntu/backup_media.sh として設置:

    #!/bin/bash
aws s3 sync /home/ubuntu/plofolia/sol/uploads/ \
  s3://plofolia-media/media-backups/ \
  --delete \
  --storage-class STANDARD_IA

echo "[$(date)] Media backup done"

    cron設定（EC2上で crontab -e）¶

    # DBバックアップ: 毎日深夜3時
0 3 * * * /home/ubuntu/backup_db.sh >> /var/log/backup_db.log 2>&1

# メディアバックアップ: 毎日深夜4時
0 4 * * * /home/ubuntu/backup_media.sh >> /var/log/backup_media.log 2>&1

    Phase 2（推奨）: WALアーカイビング（PITR対応）¶

    pg_dump では「昨日3時の状態」にしか戻れないが、WALアーカイビングを有効にすると任意の時点（分単位）に復元できる。

    例: 「今日14:32に誤って全投稿を削除してしまった」→ 14:31の状態に復元可能

    postgresql.conf に追加（EC2上）¶

    sudo nano /etc/postgresql/16/main/postgresql.conf

    archive_mode = on
archive_command = 'aws s3 cp %p s3://plofolia-media/wal-archive/%f'
archive_timeout = 300   # 最大5分でWALを強制アーカイブ

    sudo systemctl reload postgresql

    ベースバックアップ（週1回・WALの起点）¶

    # crontab に追加
0 2 * * 0 pg_basebackup -h localhost -U plofolia -D /tmp/pgbase -Ft -z \
  && aws s3 cp /tmp/pgbase/base.tar.gz s3://plofolia-media/pgbase/base_$(date +%Y%m%d).tar.gz \
  && rm -rf /tmp/pgbase

    DB復元手順¶

    通常のリストア（pg_dump から）¶

    # S3から最新バックアップをダウンロード
aws s3 ls s3://plofolia-media/db-backups/ | sort | tail -5  # 一覧確認
aws s3 cp s3://plofolia-media/db-backups/plofolia_backup_YYYYMMDD_hhmmss.sql.gz /tmp/

# DBを空にしてリストア
PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia \
  -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
gunzip -c /tmp/plofolia_backup_YYYYMMDD_hhmmss.sql.gz \
  | PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia

    PITRリストア（任意時点に戻す）¶

    # 1. PostgreSQLを停止
sudo systemctl stop postgresql

# 2. データディレクトリをバックアップ
sudo mv /var/lib/postgresql/16/main /var/lib/postgresql/16/main.bak

# 3. ベースバックアップを展開
aws s3 cp s3://plofolia-media/pgbase/base_YYYYMMDD.tar.gz /tmp/
sudo mkdir -p /var/lib/postgresql/16/main
sudo tar -xzf /tmp/base_YYYYMMDD.tar.gz -C /var/lib/postgresql/16/main
sudo chown -R postgres:postgres /var/lib/postgresql/16/main

# 4. リカバリ設定（postgresql.conf に追記）
# restore_command = 'aws s3 cp s3://plofolia-media/wal-archive/%f %p'
# recovery_target_time = '2026-04-29 14:31:00'  # 復元したい時刻

# 5. PostgreSQLを起動（リカバリモードで自動実行）
sudo systemctl start postgresql

    メディアファイルの復元¶

    aws s3 sync s3://plofolia-media/media-backups/ \
  /home/ubuntu/plofolia/sol/uploads/

    バックアップ確認コマンド¶

    # DBバックアップ一覧（S3）
aws s3 ls s3://plofolia-media/db-backups/ | sort | tail -10

# メディアバックアップのサイズ確認
aws s3 ls s3://plofolia-media/media-backups/ --recursive --human-readable --summarize | tail -2

# WALアーカイブ確認
aws s3 ls s3://plofolia-media/wal-archive/ | wc -l

# バックアップログ確認
tail -20 /var/log/backup_db.log
tail -20 /var/log/backup_media.log

    バックアップとレプリケーションの違い¶

    バックアップ レプリケーション
    目的 誤削除・破損からの復元 サーバー障害時の継続稼働
    データ 過去の任意時点に戻せる 常に最新の状態を維持
    復元時間 数分〜数十分 数十秒〜数分
    実装フェーズ Phase 1（今すぐ） Phase 3（本格稼働期）

    両方組み合わせることで完全な可用性が実現できる。

    12. 招待コードの運用¶

    現在は招待制ベータ（INVITE_CODE_REQUIRED=true）で稼働中。 有効な招待コードがないとユーザー登録できない。

    招待コードの発行手順¶

    Aurora（管理画面）から発行する（推奨）:

    1. https://aurora.plofolia.com にアクセス（admin アカウントでログイン）

    2. 左メニューの「招待コード」をクリック → 「新規発行」

    3. 設定項目:

    項目 説明 例
    ラベル 誰に渡すかのメモ（任意） Discord-beta
    コードを指定 空欄でランダム生成。指定する場合は英数字 DISCORD2025
    最大使用回数 空欄で無制限。人数を絞る場合に指定 10
    有効期限 空欄で無期限。期間限定配布の場合に指定 2026-05-31T23:59
    1. 「発行する」をクリック → コードをコピーして配布

    DB から直接発行する場合（緊急時）:

    PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "
INSERT INTO invite_codes (id, code, label, max_uses, used_count, is_active, created_at, updated_at)
VALUES (gen_random_uuid(), 'DISCORD2025', 'Discord beta', 50, 0, true, NOW(), NOW());
"

    招待コードの確認・無効化¶

    -- 有効なコード一覧
SELECT code, label, used_count, max_uses, expires_at, is_active
FROM invite_codes WHERE is_active = true ORDER BY created_at DESC;

-- コードを無効化
UPDATE invite_codes SET is_active = false WHERE code = 'XXXXX';

-- 使用済みカウントをリセット
UPDATE invite_codes SET used_count = 0 WHERE code = 'XXXXX';

    招待制を解除する場合¶

    # EC2 上の sol/.env を編集
ssh ubuntu@13.115.254.4
nano ~/plofolia/sol/.env
# INVITE_CODE_REQUIRED=true → INVITE_CODE_REQUIRED=false に変更

pm2 restart plofolia-sol --update-env

    フロントエンド側の入力欄も非表示にするには terra/.env.local の NEXT_PUBLIC_INVITE_CODE_REQUIRED を false にして npm run build が必要。

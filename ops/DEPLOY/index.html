    AWS デプロイ手順（実地記録）¶

    EC2 t3.micro + Ubuntu 22.04 LTS + PostgreSQL同居構成。 実際に踏んだ手順と詰まりポイントをまとめたもの。

    現在の本番状態（2026-04-28時点）¶

    項目 状態
    URL https://plofolia.com（HTTPS稼働中）
    EC2 i-04f92f1a7aad49ef2（ap-northeast-1a）
    IP 13.115.254.4
    SSL証明書 Let's Encrypt（2026-07-26まで有効、自動更新設定済み）
    DNS plofolia.com / www.plofolia.com → 13.115.254.4
    PM2 plofolia-sol / plofolia-nova / plofolia-terra / plofolia-aurora 全稼働

    コード更新時の再デプロイ（日常運用）¶

    → 詳細な日常運用手順は OPERATIONS.md を参照。
    → トラブル発生時は RUNBOOK.md を参照。

    方法1: ワンコマンドデプロイ（推奨）¶

    # ローカル（PowerShell）から実行するだけ
.\titan\deploy.ps1

    初回のみ E:\DEV\plofolia\ec2_key が自動生成される。以降は同じキーを再利用。

    スクリプトが自動で行うこと:

    1. EC2に一時公開鍵をプッシュ（EC2 Instance Connect、60秒有効）
    2. git push origin develop
    3. EC2上で git fetch && git reset --hard origin/develop
    4. 変更があったサービスのみビルド・PM2再起動（terra は --update-env 付き）

    方法2: EC2に直接SSHして実行¶

    # Bash から接続
ssh -i /e/DEV/plofolia/ec2_key -o StrictHostKeyChecking=no ubuntu@13.115.254.4
cd ~/plofolia && ./titan/deploy.sh

    接続前に PowerShell で公開鍵のプッシュが必要（詳細は OPERATIONS.md 参照）。

    deploy.sh の動作¶

    titan/deploy.sh は変更されたフォルダを自動検出してビルド・PM2再起動まで行う。

    重要: バックエンド（sol）の変更は npm run build でTypeScriptをコンパイルし、npm run db:migrate でDBマイグレーションを適用してから pm2 restart plofolia-sol すること。 titan/deploy.sh はビルド後に自動で npm run db:migrate を実行するため、手動実行時も忘れずに。

    前提条件¶

    • AWS EC2 インスタンス（t3.micro, Ubuntu 22.04 LTS）が起動済み
    • セキュリティグループで TCP 22 / 80 / 443 を開放済み
    • .pem キーファイルをローカルに保存済み

    サービス構成¶

    サービス名（PM2） フォルダ ポート
    plofolia-sol sol/ 4000
    plofolia-nova sol/dist/admin-app.js 4001
    plofolia-terra terra/ 3000
    plofolia-aurora aurora/ 3001

    1. SSH接続（Mac）¶

    chmod 400 ~/Downloads/plofolia-key.pem
ssh -i ~/Downloads/plofolia-key.pem ubuntu@<EC2のパブリックIP>

    2. サーバー初期セットアップ¶

    システム更新¶

    sudo apt-get update -y && sudo apt-get upgrade -y

    ダイアログが出たら Tab で <Ok> を選んで Enter

    Node.js 20 インストール¶

    curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash - && sudo apt-get install -y nodejs

    PM2 / Nginx / PostgreSQL インストール¶

    sudo npm install -g pm2 && sudo apt-get install -y nginx postgresql postgresql-contrib && sudo systemctl enable nginx && sudo systemctl enable postgresql

    PM2ログディレクトリ作成¶

    sudo mkdir -p /var/log/pm2 && sudo chown ubuntu:ubuntu /var/log/pm2

    3. PostgreSQL セットアップ¶

    sudo -u postgres psql

    CREATE USER plofolia WITH PASSWORD 'パスワード';
CREATE DATABASE plofolia OWNER plofolia;
\q

    パスワードは自分で決めて sol/.env の DATABASE_URL と揃えること

    4. コードをクローン¶

    GitHubのPersonal Access Token（PAT）が必要（パスワード認証は不可）。

    git clone https://plofolia-dev:<PAT>@github.com/plofolia-dev/plofolia.git /home/ubuntu/plofolia

    PATの発行: GitHub → Settings → Developer settings → Personal access tokens → repo スコープにチェック

    5. 環境変数設定¶

    sol（メインバックエンド）¶

    cd /home/ubuntu/plofolia/sol
cp .env.example .env
nano .env

    必須項目：

    NODE_ENV=production
DATABASE_URL=postgresql://plofolia:<パスワード>@localhost:5432/plofolia
JWT_SECRET=<openssl rand -hex 64 で生成>
FRONTEND_URL=https://plofolia.com

# ストレージ（現在はローカル保存。S3移行は有料ユーザー発生後）
STORAGE_PROVIDER=local
UPLOAD_DIR=./uploads

# 管理サーバー
ADMIN_JWT_SECRET=<openssl rand -hex 64 で別途生成>
ADMIN_PORT=4001
ADMIN_FRONTEND_URL=http://<EC2のパブリックIP>:3001

    JWT_SECRET と ADMIN_JWT_SECRET は別々に openssl rand -hex 64 を2回実行して生成すること

    terra（メインフロントエンド）¶

    cd /home/ubuntu/plofolia/terra
cp .env.example .env.local
nano .env.local

    NEXT_PUBLIC_API_URL=http://<EC2のパブリックIP>/api/v1

    aurora（管理フロントエンド）¶

    cd /home/ubuntu/plofolia/aurora
cp .env.local.example .env.local
nano .env.local

    ADMIN_API_URL=http://localhost:4001

    6. sol セットアップ（DB初期化・ビルド）¶

    cd /home/ubuntu/plofolia/sol
npm install
npm run db:generate
npm run db:migrate
npm run db:seed
npm run build

    8. terra ビルド¶

    cd /home/ubuntu/plofolia/terra
npm install
npm run build

    5〜10分かかる

    9. aurora ビルド¶

    cd /home/ubuntu/plofolia/aurora
npm install
npm run build

    10. PM2で起動（4サービス）¶

    --cwd を明示して各サービスを個別に起動する：

    pm2 start /home/ubuntu/plofolia/sol/dist/app.js --name plofolia-sol --cwd /home/ubuntu/plofolia/sol
pm2 start /home/ubuntu/plofolia/sol/dist/admin-app.js --name plofolia-nova --cwd /home/ubuntu/plofolia/sol
pm2 start /home/ubuntu/plofolia/terra/node_modules/.bin/next --name plofolia-terra --cwd /home/ubuntu/plofolia/terra -- start -p 3000
pm2 start /home/ubuntu/plofolia/aurora/node_modules/.bin/next --name plofolia-aurora --cwd /home/ubuntu/plofolia/aurora -- start -p 3001
pm2 save

    確認：

    pm2 status
# plofolia-sol / plofolia-nova / plofolia-terra / plofolia-aurora が online になっていればOK

    注意: --cwd を省略すると Next.js が .next ディレクトリを見つけられず起動に失敗する。必ず指定すること。

    PM2 自動起動設定（サーバー再起動時）¶

    pm2 startup
# 表示された sudo env PATH=... のコマンドをコピーして実行
pm2 save

    環境変数を更新して再起動する場合¶

    pm2 restart plofolia-sol --update-env

    --update-env を付けないと .env の変更が反映されない

    11. Nginx設定¶

    sudo cp /home/ubuntu/plofolia/titan/nginx/plofolia.conf /etc/nginx/sites-available/plofolia
sudo sed -i 's/your-domain.com/<ドメインまたはEC2のパブリックIP>/g' /etc/nginx/sites-available/plofolia
sudo ln -s /etc/nginx/sites-available/plofolia /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

    12. 動作確認¶

    http://<EC2のパブリックIP>/en/timeline     → メインアプリ（terra）
http://<EC2のパブリックIP>/health          → {"status":"ok"}（sol）
http://<EC2のパブリックIP>:4001/health     → {"status":"ok"}（nova）
http://<EC2のパブリックIP>:3001            → 管理画面（aurora）

    13. DNS設定（お名前.com）¶

    ドメイン: plofolia.com / EC2 IP: 13.115.254.4

    お名前.com Navi → ドメイン → DNS設定/転送設定 → DNSレコード設定を利用する

    ネームサーバー設定（「ご自身でDNSレコード設定を行う場合」を選択）:

    01.dnsv.jp / 02.dnsv.jp / 03.dnsv.jp / 04.dnsv.jp

    Aレコード追加:

    ホスト名 TYPE VALUE TTL
    （空欄） A 13.115.254.4 3600
    www A 13.115.254.4 3600

    「レコード情報の登録とあわせてDNSレコード設定用のネームサーバーに変更する」にチェックを入れて保存。

    注意: 設定後に必ず「設定する」ボタンを押すこと。押し忘れると反映されない。

    反映確認（数時間後）:

    nslookup plofolia.com 8.8.8.8
# → 13.115.254.4 になればOK

    14. HTTPS化（Certbot / Let's Encrypt）¶

    DNS反映確認後に実施。

    # Certbot インストール
sudo apt install certbot python3-certbot-nginx -y

# Nginxのserver_nameをドメインに更新してからCertbot実行
sudo sed -i 's/your-domain.com/plofolia.com/g' /etc/nginx/sites-available/plofolia
sudo nginx -t && sudo systemctl reload nginx

# SSL証明書取得（Nginxに自動設定）
sudo certbot --nginx -d plofolia.com -d www.plofolia.com

# 自動更新テスト
sudo certbot renew --dry-run

    証明書は90日有効。certbot renew はcronで自動実行される（インストール時に設定済み）。

    確認: https://plofolia.com でブラウザの鍵マークが出ればOK。

    注意: sed でserver_nameを更新する前にCertbotを実行すると「No such authorization」エラーになる。必ずNginxのserver_nameを更新してからCertbotを実行すること。

    15. 環境変数をHTTPS用に更新¶

    HTTPS化後、各サービスのURLをhttpsに更新して再ビルド・再起動：

    # sol/.env
FRONTEND_URL=https://plofolia.com
ADMIN_FRONTEND_URL=https://plofolia.com

# terra/.env.local
NEXT_PUBLIC_API_URL=https://plofolia.com/api/v1

    # terra 再ビルド
cd /home/ubuntu/plofolia/terra && npm run build && pm2 restart plofolia-terra

# sol 再起動（環境変数を反映）
pm2 restart plofolia-sol --update-env

    16. AWS S3 セットアップ（将来対応）¶

    現状: STORAGE_PROVIDER=local でサーバーローカルに保存中。 S3 移行は有料ユーザーが発生し収益が見込めた段階で実施予定。 移行時は sol/.env の STORAGE_PROVIDER=s3 に変更し下記を設定する。

    メディアファイルをS3に保存する（ローカル保存はサーバー障害時に消失するリスクがある）。

    S3バケット作成¶

    1. AWSコンソール → S3 → 「バケットを作成」
    2. バケット名: plofolia-media、バケットタイプ: 汎用
    3. リージョン: ap-northeast-1（東京）
    4. 「パブリックアクセスをすべてブロック」→ チェックを外す
    5. 警告のチェックボックスにチェック → 「バケットを作成」

    バケットポリシー設定¶

    S3 → plofolia-media → 「アクセス許可」→「バケットポリシー」→「編集」:

    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::plofolia-media/*"
    }
  ]
}

    IAMユーザー作成¶

    1. IAM → ユーザー → 「ユーザーを作成」
    2. ユーザー名: plofolia-s3
    3. 「ポリシーを直接アタッチ」→ AmazonS3FullAccess
    4. 作成後 → セキュリティ認証情報 → アクセスキーを作成 → 「アプリケーションの外部で実行されるコード」
    5. アクセスキーIDとシークレットをメモ（この画面でしか確認できない）

    DB マイグレーション適用（EC2）¶

    スキーマ変更を本番に反映するときは、生成済みのSQLファイルを直接適用する。

    # EC2 上での実行例（最新マイグレーションを適用）
PGPASSWORD=<pw> psql -h localhost -U plofolia -d plofolia \
  -f ~/plofolia/sol/src/db/migrations/0033_xxxx.sql

    まとめて適用する場合：

    for f in ~/plofolia/sol/src/db/migrations/*.sql; do
  echo "Applying $f..."
  PGPASSWORD=<pw> psql -h localhost -U plofolia -d plofolia -f "$f"
done

    コード更新時（再デプロイ）¶

    titan/deploy.sh を使うと自動で変更検出・ビルド・再起動まで行う：

    cd /home/ubuntu/plofolia
./titan/deploy.sh

    手動で行う場合：

    cd /home/ubuntu/plofolia
git pull

# sol/nova 変更あり
cd sol && npm run build
pm2 restart plofolia-sol plofolia-nova

# terra 変更あり
cd ../terra && npm run build && pm2 restart plofolia-terra

# aurora 変更あり
cd ../aurora && npm run build && pm2 restart plofolia-aurora

    トラブルシューティング¶

    502 Bad Gateway¶

    # 1. PM2の状態確認
pm2 list

# 2. ログ確認
pm2 logs --lines 50 --nostream

# 3. ポートが実際にlistenしているか確認
curl -s http://127.0.0.1:4000/health
curl -s http://127.0.0.1:3000

# 4. Nginx設定確認（proxy_pass先のポートが正しいか）
sudo cat /etc/nginx/sites-enabled/plofolia

# 5. Nginxエラーログ確認
sudo tail -50 /var/log/nginx/error.log

    よくある原因:

    • PM2サービスが停止している → pm2 restart all
    • pm2 start 時に --cwd を指定していない → Next.jsが .next を見つけられず起動失敗
    • Nginxのproxy_passポートが間違っている（solは4000、terraは3000）
    • HTTPS化後に環境変数（FRONTEND_URL等）が古いhttpのままでCORSエラー
    • .env 変更後に --update-env なしで再起動 → 環境変数が反映されない

    DBパスワード認証エラー¶

    sol/.env の DATABASE_URL のパスワードとPostgreSQLのユーザーパスワードが一致しているか確認。

    sudo -u postgres psql -c "ALTER USER plofolia WITH PASSWORD '新しいパスワード';"

    S3アップロードエラー（NoSuchBucket）¶

    sol/.env の S3_BUCKET_NAME が正しいか確認。変更した場合は --update-env で再起動：

    pm2 restart plofolia-sol --update-env

    ADMIN_JWT_SECRET is not set で nova が起動しない¶

    sol/.env に ADMIN_JWT_SECRET が設定されているか確認。openssl rand -hex 64 で生成して設定する。

    next.config.ts エラー¶

    このバージョンのNext.jsは .ts 拡張子の設定ファイルに未対応。next.config.mjs を使うこと（リポジトリ修正済み）。

    float is not a function エラー¶

    drizzle-orm v0.31 では float が廃止。doublePrecision を使うこと（リポジトリ修正済み）。

    PM2ログフォルダエラー¶

    sudo mkdir -p /var/log/pm2 && sudo chown ubuntu:ubuntu /var/log/pm2

    GitHubクローン時にユーザー名/パスワードを聞かれる¶

    GitHubはパスワード認証不可。PATをURLに埋め込む：

    git clone https://plofolia-dev:<PAT>@github.com/plofolia-dev/plofolia.git

    Certbot「No such authorization」エラー¶

    Nginxの server_name がドメインと一致していない。先にNginx設定を更新してからCertbotを実行すること。

    DNS「SERVFAIL」エラー¶

    お名前.comでAレコードを設定後、「設定する」ボタンを押し忘れている可能性がある。設定画面を再確認すること。

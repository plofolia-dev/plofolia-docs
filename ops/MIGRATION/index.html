    Plofolia 移行手順書¶

    インフラ構成を変更する際の手順をまとめたドキュメント。
    各移行は本番への影響が大きいため、必ず事前にローカルで検証してから実施すること。

    目次¶

    1. インスタンスタイプ変更（t3.micro → 上位）
    2. EC2 インスタンス移行（新サーバーへの引っ越し）
    3. ストレージ移行（local → S3）
    4. DB 移行（EC2 ローカル → RDS）
    5. 将来の構成変更（参考）

    現在のインフラ構成（移行前）¶

    インターネット
    │
    ▼
EC2 t3.micro (13.115.254.4, ap-northeast-1a)
    │
    ├── Nginx (リバースプロキシ)
    │     ├── :443 → plofolia-terra  (:3000)
    │     ├── :443/api/v1 → plofolia-sol  (:4000)
    │     ├── :443/admin → plofolia-nova  (:4001)
    │     └── :443/aurora.plofolia.com → plofolia-aurora  (:3001)
    │
    ├── PostgreSQL 16 (EC2 ローカル)
    ├── Redis 7 (EC2 ローカル / 将来的に実装)
    └── メディアファイル (EC2 ローカル: ~/plofolia/sol/uploads/)

S3: plofolia-media
    ├── backups/db/           DBバックアップ（日次）
    ├── media-backups/        メディアファイルバックアップ（日次）
    └── wal-archive/          WAL アーカイブ（有効化時）

    1. インスタンスタイプ変更（t3.micro → 上位）¶

    ユーザー増加でメモリ・CPU が逼迫したときに実施。ダウンタイムが発生する（5〜10 分）。

    タイミングの目安¶

    症状 推奨アクション
    free -h で available が常に 100MB 以下 t3.small（2GB）へ移行
    CPU が常時 80% 以上 t3.medium（2コア・4GB）へ移行
    OOM Kill が週 1 回以上発生 t3.small へ移行 + swap を 2GB 確保

    コスト目安（ap-northeast-1、オンデマンド）:

    タイプ vCPU メモリ 月額（USD）
    t3.micro 2 1 GB ~$8
    t3.small 2 2 GB ~$16
    t3.medium 2 4 GB ~$32

    手順¶

    # 1. 事前準備: バックアップを取る
./titan/backup-db.sh

# 2. メンテナンス告知（必要であれば）

# 3. EC2 インスタンスを停止（AWS コンソール）
# EC2 → インスタンス → 停止

# 4. インスタンスタイプを変更（AWS コンソール）
# EC2 → インスタンス → アクション → インスタンスの設定 → インスタンスタイプを変更

# 5. インスタンスを起動

# 6. SSH で接続して動作確認
pm2 status
curl -s http://127.0.0.1:4000/health

# 7. swap を調整（2GB に増量する場合）
free -h
# swap が 0 または 1G なら 2G に再作成
sudo swapoff /swapfile
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

    注意: t3.micro → t3.small はストレージもセキュリティグループもそのまま引き継がれる。
    インスタンス停止中は料金が発生しないが、EBS（ストレージ）は課金され続ける。

    2. EC2 インスタンス移行（新サーバーへの引っ越し）¶

    現在の EC2 を廃止して新しいインスタンスを立てる場合。
    リージョン変更・OS アップグレード・アーキテクチャ変更（arm64 など）が主なユースケース。

    推奨手順: AMI スナップショットから新インスタンスを起動¶

    # 1. 最新 AMI が存在することを確認（毎週日曜 AM3:30 に自動作成）
# AWS コンソール → EC2 → AMI → 「plofolia-ami-YYYYMMDD」を確認

# 2. 万が一のため手動でも AMI を作成
# EC2 → インスタンス → アクション → イメージとテンプレート → イメージの作成
# 名前: plofolia-manual-YYYYMMDD
# 「再起動しない」にチェックを入れると無停止でスナップショットを取得できる（整合性は下がる）

    # 3. 新インスタンスを起動（AWS コンソール）
# AMI → 選択した AMI → 「AMI からインスタンスを起動」
# - タイプ: 目的に応じて選択
# - キーペア: 既存の plofolia キーペアを選択
# - セキュリティグループ: 既存のものを適用（SSH:22, HTTP:80, HTTPS:443 のみ）
# - ストレージ: 既存と同容量以上

# 4. 新インスタンスに SSH 接続
NEW_IP="新インスタンスのIP"
ssh -i E:\DEV\plofolia\ec2_key ubuntu@$NEW_IP

# 5. 動作確認
pm2 status
curl -s http://127.0.0.1:4000/health
PGPASSWORD=polotiaakplt psql -h localhost -U plofolia -d plofolia -c "SELECT COUNT(*) FROM users;"

    DNS 切り替え（Route 53 を使っている場合）¶

    # AWS コンソール → Route 53 → ホストゾーン → plofolia.com
# A レコードを新インスタンスの IP に変更
# TTL が短い（300秒）場合は 5 分で反映

    Nginx の IP 設定（Elastic IP 未使用の場合）¶

    現在 EIP 未使用のため、IP が変わると DNS を更新する必要がある。

    # 新インスタンスで Nginx 設定を確認（IP ベースの設定がないか）
sudo cat /etc/nginx/sites-available/plofolia | grep -i "server_name\|ip"
# server_name でドメイン指定しているだけなら変更不要

    旧インスタンスの停止¶

    移行後 1〜2 日は旧インスタンスを停止状態で保持し、問題なければ終了（terminate）する。

    3. ストレージ移行（local → S3）¶

    現在メディアファイルは EC2 ローカル（~/plofolia/sol/uploads/）に保存している。
    ユーザー数が増えてディスクが逼迫したとき、または EC2 を複数台に増やすときに実施。

    移行前の確認¶

    # 現在のメディアサイズ
du -sh ~/plofolia/sol/uploads/
# S3 バケット（plofolia-media）へのアクセス確認
aws s3 ls s3://plofolia-media/

    Step 1: 既存メディアを S3 にアップロード¶

    # すべてのメディアを S3 に同期（uploads/ 直下の構造を維持）
aws s3 sync ~/plofolia/sol/uploads/ s3://plofolia-media/uploads/ \
  --region ap-northeast-1 \
  --storage-class STANDARD_IA

# アップロード確認
aws s3 ls s3://plofolia-media/uploads/ --recursive --summarize | tail -3

    Step 2: sol の設定を変更¶

    sol/.env を編集:

    nano ~/plofolia/sol/.env

    変更箇所:

    # 変更前
STORAGE_PROVIDER=local

# 変更後
STORAGE_PROVIDER=s3
AWS_S3_BUCKET=plofolia-media
AWS_S3_PREFIX=uploads
AWS_REGION=ap-northeast-1
# AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY は既存の値を使用

    注意: S3 移行前に sol/src/services/storage.ts の S3 対応コードが実装済みであることを確認すること。現在は local のみ対応。S3 対応は将来の実装タスク。

    Step 3: sol を再起動して動作確認¶

    pm2 restart plofolia-sol --update-env

# 新規アップロードのテスト
# → ブラウザで画像・動画を投稿し、S3 に保存されるか確認
aws s3 ls s3://plofolia-media/uploads/ --recursive | tail -5

    Step 4: CloudFront CDN の設定（任意・推奨）¶

    S3 に移行したら CloudFront を前段に置くとパフォーマンスが向上する。

    # AWS コンソール → CloudFront → ディストリビューション作成
# オリジン: plofolia-media.s3.ap-northeast-1.amazonaws.com
# キャッシュポリシー: CachingOptimized（画像・動画向け）
# 代替ドメイン: media.plofolia.com（任意）

# Nginx で /api/v1/uploads → CloudFront に向け直す
sudo nano /etc/nginx/sites-available/plofolia
# location /api/v1/uploads/ {
#   proxy_pass https://dXXXX.cloudfront.net/uploads/;
# }

    4. DB 移行（EC2 ローカル → RDS）¶

    EC2 上の PostgreSQL を Amazon RDS に移行する。
    HA 構成・自動バックアップ・フェイルオーバーが必要になったタイミングで実施。

    実施の目安: MAU 10,000 人超え、または可用性 SLA を設けるとき。

    コスト目安（RDS PostgreSQL、ap-northeast-1）¶

    クラス vCPU メモリ ストレージ 月額（USD）
    db.t3.micro 2 1 GB 20 GB gp2 ~$25
    db.t3.small 2 2 GB 20 GB gp2 ~$50
    db.t3.medium 2 4 GB 100 GB gp2 ~$100

    Step 1: RDS インスタンスを作成（AWS コンソール）¶

    RDS → データベースの作成
- エンジン: PostgreSQL 16
- テンプレート: 開発/テスト（本番は「本番稼働用」）
- DB 識別子: plofolia-db
- マスターユーザー: plofolia
- マスターパスワード: （新しい強力なパスワードを設定）
- インスタンスクラス: db.t3.micro から始める
- ストレージ: 20 GB gp2（自動スケーリング有効）
- VPC: EC2 と同じ VPC
- セキュリティグループ: EC2 からの 5432 ポートのみ許可
- 自動バックアップ: 7日間保持
- パラメータグループ: デフォルト

    # RDS のエンドポイントを確認（作成後 AWS コンソールで表示）
RDS_ENDPOINT="plofolia-db.xxxxxxxx.ap-northeast-1.rds.amazonaws.com"

    Step 2: 既存 DB を RDS に移行¶

    # EC2 上で実行

# 1. ダンプを取得（サービスは停止しない — pg_dump はオンラインで動作する）
PGPASSWORD=polotiaakplt pg_dump -h localhost -U plofolia -d plofolia \
  --no-owner --no-acl \
  | gzip > /tmp/plofolia_rds_migration.sql.gz

# 2. RDS に DB とユーザーを作成
PGPASSWORD=新RDSマスターパスワード psql \
  -h $RDS_ENDPOINT -U plofolia -d postgres << 'EOF'
CREATE DATABASE plofolia;
EOF

# 3. RDS にリストア
gunzip -c /tmp/plofolia_rds_migration.sql.gz \
  | PGPASSWORD=新RDSマスターパスワード psql \
    -h $RDS_ENDPOINT -U plofolia -d plofolia

# 4. データ確認
PGPASSWORD=新RDSマスターパスワード psql \
  -h $RDS_ENDPOINT -U plofolia -d plofolia \
  -c "SELECT COUNT(*) FROM users; SELECT COUNT(*) FROM posts;"

    Step 3: sol の接続先を切り替え¶

    # sol/.env を更新（EC2 上）
nano ~/plofolia/sol/.env

    # 変更前
DATABASE_URL=postgresql://plofolia:polotiaakplt@localhost:5432/plofolia

# 変更後
DATABASE_URL=postgresql://plofolia:新RDSパスワード@plofolia-db.xxxxxxxx.ap-northeast-1.rds.amazonaws.com:5432/plofolia

    # サービスを停止→再起動（切り替えのダウンタイム：数秒）
pm2 stop plofolia-sol plofolia-nova
pm2 start plofolia-sol plofolia-nova --update-env
pm2 status

# 接続確認
curl -s http://127.0.0.1:4000/health
pm2 logs plofolia-sol --lines 20 --nostream

    Step 4: EC2 上の PostgreSQL を停止¶

    切り替え後 1〜2 日動作確認してから:

    # EC2 上の PostgreSQL を停止（RDS に移行済みのため不要）
sudo systemctl stop postgresql
sudo systemctl disable postgresql

# バックアップスクリプトの接続先も更新
nano ~/titan/backup-db.sh
# -h localhost → -h $RDS_ENDPOINT に変更

    Step 5: バックアップ戦略の更新¶

    RDS は自動バックアップ（7日間）を持っているため、backup-db.sh による pg_dump は任意となる。
    ただし S3 への保存も継続するとより安全:

    # backup-db.sh の接続先を RDS エンドポイントに変更
nano ~/titan/backup-db.sh

    5. 将来の構成変更（参考）¶

    現段階では不要だが、本格稼働期を見越した構成変更メモ。

    CloudFront 導入（CDN）¶

    • 対象: 静的アセット（画像・動画・Next.js チャンク）
    • 目的: 配信高速化・帯域コスト削減
    • タイミング: MAU 5,000 人超え、または S3 転送コストが月 $10 を超えたとき

    ALB + 複数 EC2（水平スケール）¶

    • 対象: sol（API サーバー）
    • 目的: 高可用性・負荷分散
    • 前提: セッション管理を Redis に移行（現在は JWT 認証のため基本はステートレス）
    • SSE 接続の扱い: ALB はスティッキーセッションでソルバーを固定するか、Redis Pub/Sub で全インスタンスにブロードキャストする実装が必要

    ElastiCache (Redis)¶

    • 対象: セッションキャッシュ・Rate Limiting の共有カウンター
    • 現状: Redis は EC2 ローカルに存在するが、sol はまだ接続していない
    • タイミング: 複数 EC2 で Rate Limiting を共有したいとき

    RDS Multi-AZ¶

    • RDS 移行後にフェイルオーバーが必要になったとき
    • コンソールから「マルチ AZ 配置を変更」で有効化（ダウンタイムあり・数分）
